Security Policy and Vulnerability Disclosure

Adding a root `SECURITY.md` file explaining how developers and security researchers should report vulnerability is important. You don't want a security threat to be turned into a public issue (This gives you time to analyze and possibly fix before disclosure). > âš ï¸ If you are a developer, never report a security threat using a public GitHub issue. Use OpenSSF scorecard

!scorecard The OSSF scorecard initiative is really good to assess your project against security best practices. I am not the first to write about this. Configure Dependabot

It is recommended to use Dependabot for updating your dependencies and GitHub actions (yes, it also supports updating workflows in a secure way ğŸ˜). You only need to add a `.github/dependabot.yml` config file. Personally I recommend a weekly interval (with a lot of projects daily is a bit horrible). 
yml
version: 2
updates:
  - package-ecosystem: github-actions
    directory: /
    schedule:
      interval: weekly

  - package-ecosystem: npm
    directory: /
    schedule:
      interval: weekly
